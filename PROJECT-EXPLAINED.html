AI Failure Investigation System

An observability tool that watches your Large Language Model (LLM) app in production, records when its answers go wrong, and helps you figure out why — and which model to use for which job. This page explains the whole thing, A to Z, including the honest answer to “is this real data or fake?”

01 The basics

Start here if the whole thing is a mystery.

What is this project, in one sentence?

It’s a monitoring and diagnosis system for AI (LLM) apps: your app tells this system every time the AI gave a bad answer (or a good one), and the system stores those events, spots recurring problems, and shows you charts and suggestions on a dashboard.

What real problem does it solve?

When you build an app on top of GPT-4, Claude, Gemini, etc., the model sometimes fails: it makes facts up (“hallucination”), returns nothing, produces broken JSON, times out, and so on. In production you usually have no idea how often this happens, on which model, for which type of task, or why. This project is the equivalent of “error monitoring” (like Sentry/Datadog) but purpose-built for the specific ways LLMs fail.

Who is it actually for?

Engineering / ML teams running LLM features in production who want to answer questions like:

  • “What’s our real failure rate this week, per model?”
  • “Which model should we use for summarization vs. code generation, based on our traffic?”
  • “Are timeouts caused by high latency? Are hallucinations linked to poor retrieval?”
  • “Give me the exact prompts that failed so I can build a regression test set.”

What counts as a “failure” here?

The system recognises 9 named failure types. An event with no failure type recorded is treated as a success.

TypeMeaning
hallucinationMade-up / incorrect facts
empty_responseBlank or null answer
malformed_responseStructurally invalid output (e.g. broken JSON)
timeoutRequest took too long
semantic_errorLow-quality or logically wrong answer
confidence_mismatchModel was confident but wrong (or vice-versa)
retrieval_failureRAG search returned poor/no context
rate_limitedHit the provider’s rate limit
token_limitContext/output token limit exceeded

Events are also tagged with a task type (summarization, code_generation, extraction, rag_qa, classification, translation, agentic, creative_writing, chat, other) and a severity (critical / high / medium / low).

02 How it works end-to-end

Follow a single event from your app to the dashboard.

1 · Your app
Your code calls an LLM (OpenAI, Anthropic, …). Something goes right or wrong.
↓
2 · Python SDK sdk/client.py
You call report_failure(...) or report_success(...). The SDK buffers events and sends them in batches on a background thread.
↓
3 · REST API (FastAPI) POST /api/v1/events
Checks your API key, redacts PII (emails, cards, SSNs…), estimates cost from token counts, and bulk-inserts rows.
↓
4 · Database (PostgreSQL / SQLite)
Every event is stored as a structured row: model, task, latency, tokens, cost, confidence, retrieval score, failure type, etc.
↓
5 · Analysis engine pattern_engine.py + routers
Groups failures into patterns, computes correlations, ranks models per task, checks for failure-rate spikes (and can fire a Slack/webhook alert).
↓
6 · Dashboard (Streamlit) dashboard/app.py
Reads the API and draws the charts: Overview, Failures, Patterns, Models, Model Fit, Correlations.

Is there any AI/model running inside this system?

Important

No. This is a surprisingly common misconception. This project does not run any LLM itself and does not call OpenAI/Anthropic. It is a plain data pipeline: it receives structured records about LLM calls that happened in your app, stores them, and does ordinary statistics (counting, averaging, a correlation coefficient) on top. All the “intelligence” is deterministic Python and pandas — not a neural network.

What are the API endpoints?

EndpointDoes what
POST /eventsIngest a batch of events (the main entry point)
GET /failuresBrowse individual failures with filters
GET /patternsList detected failure clusters
GET /patterns/{id}/exportDownload a pattern’s failing prompts as a JSONL eval set
GET /modelsPer-model performance stats
GET /recommendationsRank models per task (reliability → cost → latency)
GET /correlationsWhich factors are associated with failures
GET /statsSystem-wide summary numbers
POST /feedbackTell the system a classification was right/wrong
GET /healthHealth check

Every route requires a Bearer API key. Keys are stored as SHA-256 hashes, and requests are rate-limited.

03 The “brain” — how the insights are actually computed

This matters a lot for judging whether to trust the output. Here’s exactly what each “smart” feature does under the hood.

How does “pattern detection” work? real computation

Despite the word “clustering” in the docs, it is not machine-learning clustering. It is a pandas groupby: failures from the last N hours are grouped by (failure_type, model_name). Any group with ≥ 3 occurrences becomes a “pattern” with counts, averages (confidence, latency, retrieval), and a first/last-seen window.

So a “pattern” = “this failure type keeps happening on this model.” That’s genuinely useful and honest — just simpler than the word implies.

Where do the remediation suggestions come from? hard-coded templates

They’re fixed text written by the author, chosen by a lookup table on failure type (with a couple of if branches on confidence/retrieval). Example: a hallucination pattern with high confidence always returns “Add fact-checking prompts or chain-of-thought verification…”.

Be clear-eyed

These are sensible generic engineering tips, not AI-generated advice tailored to your specific prompt. Two different apps with the same failure type get the exact same suggestion.

How are correlations computed? real statistics

This is genuine stats. It uses the phi coefficient (φ) — a standard measure of association between two yes/no variables — on pairs the author picked, such as:

  • retrieval_score < 0.5 vs. hallucination
  • latency > 75th percentile vs. timeout
  • confidence < 0.5 vs. hallucination / confidence_mismatch

It only runs if there are ≥ 30 events, and only reports associations with |φ| > 0.1. The maths is real; the pairs it checks are a fixed, hand-chosen list (not automatically discovered).

How are model recommendations (“Model Fit”) computed? real computation

For each task type, it groups your events by model and computes the observed failure rate, then ranks models by: reliability first → then cost per call → then latency. A model needs at least min_events (default 20) before it can be “recommended”; otherwise it’s shown but flagged as too small a sample.

This is a real, honest calculation over whatever data is in the database. Which is the crux of the next section: garbage in, garbage out.

How is cost calculated? approximate list prices

If you don’t supply an exact cost_usd, the API estimates it from token counts using a built-in table of public list prices (USD per million tokens) in backend/pricing.py. It matches your model name by longest prefix. These are ballpark public prices for ranking models by cost — not your real invoice. You can override them with the MODEL_PRICING_JSON env var.

05 The honest answer: is this real data or predefined?

This is your main question, so here it is directly and without spin.

The short version

The demo data is fake (synthetic) and is not verified against any real-world benchmark. The machinery is real and produces genuine results — but only when you feed it your own production traffic. The screenshots in the README are a plausible-looking story, not measured facts.

Where does the data in the screenshots / demo come from?

From scripts/seed_demo.py. That script invents events with Python’s random generator. It has hard-coded assumptions baked in, for example:

BASE_FAILURE_RATE = {"gpt-4o":0.06, "llama-3-70b":0.12, ...}
TASK_FAILURE_OVERRIDES = {
    ("gpt-4o","code_generation"): 0.03,      # "gpt-4o is great at code"
    ("claude-3-5-sonnet","rag_qa"): 0.03,    # "claude is great at RAG"
    ("llama-3-70b","extraction"): 0.22,      # "llama is weak at extraction"
}

Then it rolls dice against those numbers to manufacture successes and failures. The correlations (low retrieval → hallucination, high latency → timeout) are deliberately injected so the charts show something interesting.

So when the demo says “gpt-4o is best for code generation,” is that true?

No

That conclusion is true only because the seed script was written to make it true. It is a self-fulfilling number, not a measurement of the real GPT-4o. Do not cite the demo dashboard as evidence about how any real model performs. It’s a UI demonstration, nothing more.

Will I get the same result every time?

With the demo: broadly yes — the shape of the story is fixed by the script (though exact numbers wiggle unless you pass a --seed). It’s designed to always tell the same tidy narrative.

With real traffic: no — results change as your app’s real behaviour changes, which is the entire point.

Is any of this “verified” against a strong external reference?

No external benchmark, dataset, or ground truth is wired in. There is no comparison to public leaderboards (e.g. MMLU, HELM), no third-party validation, and no live model evaluation. The system doesn’t judge whether an answer is a hallucination — your app decides that and reports it. The system just aggregates what you tell it.

What is tested is the code: there’s a pytest suite + CI verifying the API, pattern engine, correlations, redaction, auth, rate-limiting, etc. behave correctly. That’s software correctness — different from “the insights are scientifically validated.”

Is the pricing table real?

They’re approximate public list prices from when the table was written, used to rank models by relative cost. Treat them as ballpark, not billing. Real prices change and depend on your contract.

Bottom line

Think of it like a fitness tracker with a demo mode. The demo shows fake step counts to prove the app works — nobody actually walked those steps. But strap it on and it measures your real steps accurately. This project is the same: the demo data is staged, the engine that measures is real. Its usefulness comes entirely from connecting it to your own live LLM traffic.

06 How to use it for real (and get trustworthy results)

The steps that turn the demo into a genuinely useful tool.

How do I run the whole thing?

# 1. configure secrets (Postgres password + API key)
cp .env.example .env

# 2. start API + dashboard + database
docker-compose up -d

# Dashboard →  http://localhost:8501
# API       →  http://localhost:8000
# API docs  →  http://localhost:8000/docs

To fill the demo (fake) data for a first look: python scripts/seed_demo.py --events 2000 --days 7.

How do I get real, trustworthy numbers?

Instrument your actual app so it reports every real LLM call — both successes and failures:

from sdk.client import FailureInvestigator

inv = FailureInvestigator(api_key="sk-...", endpoint="http://localhost:8000")

# good call
inv.report_success(model_name="gpt-4o", task_type="summarization",
                   latency_ms=740, input_tokens=1800, output_tokens=220)

# bad call — YOUR code decided this was a hallucination
inv.report_failure({
    "timestamp": "...", "model_name": "gpt-4o", "task_type": "rag_qa",
    "failure_type": "hallucination", "failure_severity": "high",
    "confidence_score": 0.3, "retrieval_score": 0.2,
})
The key insight

The quality of every chart depends entirely on you reporting successes too (not just failures) and on your own logic for deciding what’s a failure. Report only failures and your “failure rate” will look like 100%. This is the honest work the tool can’t do for you.

What about privacy — does it store my users’ prompts?

Prompts and responses can be stored, but PII redaction is on by default: emails, credit cards, SSNs, phone numbers, IPs and API keys are stripped before anything is written to the database. You can also enable data retention to auto-delete events older than N days. See SECURITY.md.

What are the genuinely useful, real features?

  • Real failure-rate tracking per model/task once you send real events.
  • Real correlation analysis (phi coefficient) surfacing e.g. “low retrieval score really does track with hallucinations in your data.”
  • Pattern → JSONL export: turn a real failure cluster into a regression test set with one click.
  • Shadow A/B: run a candidate model on a sample of live prompts and compare — the strongest evidence for “model X is better at task Y.”
  • Alerting: Slack/webhook when a new pattern appears or a failure rate spikes.

07 Honest limitations & maturity

So you can set expectations correctly.

It doesn’t detect failures for you

Your app must classify each call. The system only aggregates your labels.

“Clustering” is groupby

No ML clustering or embeddings — patterns are exact (type, model) groups.

Remediation is templated

Fixed generic tips per failure type, not tailored AI advice.

Correlations are a fixed list

Only the hand-picked factor pairs are checked, not auto-discovered ones.

Costs are estimates

Public list prices for ranking, not your actual invoice.

Single-tenant

Multi-tenant support and ML-based root cause are on the roadmap, not built.

How to describe it accurately

“A working, MIT-licensed LLM observability platform — SDK + FastAPI + Postgres + Streamlit — that ingests LLM call events, aggregates failure rates and patterns, runs real correlation analysis, and ranks models per task on your own traffic. It ships with a synthetic demo dataset for illustration; real value requires instrumenting your own app.” That sentence is fully defensible.

Generated from a full read of the codebase (backend routers, pattern_engine.py, correlations.py, recommendations.py, pricing.py, the SDK, and seed_demo.py). Every “real vs. synthetic” badge on this page reflects what the code actually does.
